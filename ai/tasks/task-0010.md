## Objective
Close out Milestone M3 by adding the manifest writer, the seven-branch
`ResolveRevision` resolver, and the legacy `.orun/plans/<checksum>.json`
mirror body to the existing `internal/revision` package — without touching
production callers (`cmd/orun`, `internal/state`, `internal/runner`,
`internal/runbundle` stay untouched; M5 wires them).

Specifically, ship:

1. `internal/revision/manifest.go` exposing
   - `WriteManifest(ctx, cfg Config, rev PlanRevision, trig triggerctx.TriggerOccurrence) error`
     that composes the JSON shape in `data-model.md` §4 (`RevisionManifest`)
     and writes it to `revisions/<revisionKey>/manifest.json` via
     `statestore.CreateIfAbsent` (idempotent re-runs MUST surface
     `ErrExists` from the store and let the caller decide; do not swallow).
   - `UpdateLatestExecutionSummary(ctx, cfg Config, revKey string,
     latestExecKey, latestExecStatus string) error`
     that reads `manifest.json`, mutates `summary.latestExecutionKey` and
     `summary.latestExecutionStatus`, and writes it back via
     `statestore.CompareAndSwap` (loser-retries up to `casRetryBudget`,
     reusing the existing constant in `writer.go`).
   - `JobCount` MUST come from a small `ManifestInputs` struct (or
     equivalent) passed into `WriteManifest` so the caller — currently the
     `WriteRevision` driver and later M5 — supplies it explicitly. PR-A's
     `RevSummary.JobCount = 0` placeholder must be removed from any
     persisted artifact in this PR's hot path: when `WriteRevision` is
     called by a caller that knows the job count, that count flows into
     both `RevSummary` and the manifest. Pick one of these surfaces and
     document the decision in the implementer report:
     - **Option A (preferred):** add `Config.JobCount int` and have
       `WriteRevision` thread it through `RevSummary`.
     - **Option B:** add a new `WriteRevisionInput` struct and accept it on
       `WriteRevision`. If you choose B, deprecate the bare-args form via a
       wrapper for existing tests.
     Either way, M3 callers that don't know the count yet (none in tree
     today) MUST keep working with `JobCount = 0` and the manifest MUST
     reflect that exact value.

2. `internal/revision/resolver.go` exposing
   - `ResolveRevision(ctx, cfg Config, arg string) (PlanRevision, ResolveSource, error)`
     implementing the seven-branch chain in
     `compatibility-and-migration.md` §3 in this exact order:
     1. empty arg → `refs/latest-revision.json`
     2. arg matches an existing file → load plan bytes from file; synthesize
        a `system.manual` revision **in-memory only** (no disk write).
     3. arg matches `^rev-[a-z0-9-]+-p[a-f0-9]{8}(-x\d+)?$` →
        `revisions/<arg>/plan.json` + `revision.json`.
     4. arg matches a named ref →
        `refs/named/<arg>.json` → revision key from the ref.
     5. arg is hex (legacy plan hash, regex `^[a-f0-9]{8,64}$`) →
        `.orun/plans/<arg>.json` legacy load + synthesize a migrated
        revision in-memory (`triggerType: "system"`, `triggerName:
        "system.migrated"`).
     6. arg matches a component name → return `ResolveSourceComponent`
        with `ErrComponentRunUnchanged` (a new typed sentinel scoped to
        this package; document that M5 CLI rewires this branch into the
        existing component-run path; do **not** invent a component lookup
        in M3).
     7. otherwise → typed `ErrAmbiguousArg`.
   - `ResolveSource` is an enum-ish typed string (`ResolveSourceLatest`,
     `ResolveSourceFile`, `ResolveSourceRevisionKey`, `ResolveSourceNamedRef`,
     `ResolveSourceLegacyHash`, `ResolveSourceComponent`) so M5 callers can
     branch on intent without re-parsing `arg`.
   - Reader fallback per §4: branch 3 (and only branch 3) MAY fall back via
     legacy path (`os.Stat` on `.orun/plans/<planHash>.json`) when the
     `StateStore` returns `ErrNotFound` AND the user-supplied arg shape
     allows it. **In this PR, branch 3 does NOT fall back** — only branch 5
     reads legacy paths. Document the reasoning in the implementer report.
   - The resolver MUST NOT write to disk. Read-only by contract.

3. `internal/revision/writer.go` — promote `writeCompatibilityMirror` from
   `// TODO(m5)` no-op to a real conditional write of the legacy mirror
   per `compatibility-and-migration.md` §2:
   - When `Config.CompatibilityWrites == true`:
     - Write `.orun/plans/<planHashHex>.json` byte-identical to the
       canonical `plan.json` via `statestore.CreateIfAbsent` (legacy paths
       routed through `paths.go`-style helpers — add the helper if missing,
       no string concatenation).
     - Write `.orun/plans/latest.json` byte-identical to the same bytes
       via `statestore.Write` (last-write-wins is acceptable for this
       alias per spec §2).
     - On `ErrExists` for the checksum body, treat as a no-op success
       (idempotent re-runs of `WriteRevision` for the same plan bytes).
     - On any other store error, wrap and return.
   - When `Config.CompatibilityWrites == false`: return nil; do not write
     anything; cover both branches in tests.
   - The `<planHashHex>` segment is the `planHash` minus the `sha256:`
     prefix (data-model.md §3.1). Centralize the strip in a helper and unit
     test it.

4. Tests (all in `internal/revision/`):
   - `manifest_test.go` — happy path, idempotent `WriteManifest`,
     `UpdateLatestExecutionSummary` CAS contention (force a conflicting
     write between Read and CAS, assert the retry succeeds inside
     `casRetryBudget`, assert eventual `ErrConflict` after budget),
     JSON byte-stability via `internal/testfx/statefs.AssertJSONFile`.
   - `resolver_test.go` — **all seven branches** + ambiguity error +
     legacy hex fallback round-trip. Use the `internal/testfx/statefs`
     workspace harness so each branch test is self-contained.
   - `writer_compat_test.go` — `CompatibilityWrites=true` writes both
     `.orun/plans/<hash>.json` and `.orun/plans/latest.json` byte-identical
     to `plan.json`; `CompatibilityWrites=false` writes neither; idempotent
     re-run on the same plan succeeds with no error.
