## Objective

Implement `orun github runs --details` Level 2 manifest download so users can get exact shard status from remote manifest files without full hydration or log download. This satisfies GitHub Artifacts Requirement 11 and the Phase 9 design's three-level artifact detail model.

## PR Boundary

1. Add a GitHub artifact manifest-only download/read path.
   - Prefer a small helper in `internal/artifactstore/github` that downloads an artifact ZIP to a temp directory and returns only the parsed `manifest.json` / `RunBundleShardManifest`.
   - Reuse existing authenticated client, retry, ZIP path traversal defense, and shard validation where practical.
2. Wire `cmd/orun/command_github.go` so `orun github runs --details` uses Level 2 detail mode.
   - Default `orun github runs` must remain Level 1: list artifacts and parse names only.
   - `--details` may download manifests for Orun artifacts and use manifest status/role/exec-id/job metadata for more accurate summary output.
3. Add focused unit/integration tests for the Level 2 path.
   - Cover `--details` behavior with mocked GitHub API/artifact ZIP data or a small seam around the manifest detail helper.
   - Preserve existing `github logs`, `github pull`, and `github status` tests.
4. Update GitHub Artifacts specs/docs only if the implementation changes the documented behavior or completes the Requirement 11 current-status row.

Explicit scope boundary: this is a CLI remote-inspection PR only. It must not introduce the E2E workflow, TUI cockpit work, root workflow template decisions, or broad GitHub Artifacts refactors.
