## Objective

Rewire `orun status`, `orun logs`, `orun describe`, and `orun get plans`
to read execution and revision state from the canonical revision-first
layout (`revisions/<revKey>/executions/<execKey>/{execution,
snapshot.latest}.json`, `refs/latest-execution.json`,
`indexes/executions/`, `indexes/triggers/`, `refs/named/`), with the
existing `.orun/executions/<id>/` layout as a transparent fallback path
exactly as `compatibility-and-migration.md` §3–§4 prescribe. Surface
`bridge-mirror-failed` events on stderr (best-effort, never block) so
operators have a signal when the bridge silently degraded under M4
contract. Expose the new triplet shape (revisionKey + executionKey +
legacyExecID) in `orun describe` output, and add the new flags / aliases
listed in `cli-surface.md` §3.2 / §5.1 / §6.1.

## PR Boundary

In scope:

1. `cmd/orun/command_status.go`
   - Default path: read `refs/latest-execution.json` via the
     `internal/executionstate` resolver. On miss, scan
     `.orun/executions/` and synthesize a display row from the most
     recent legacy run (compat fallback, cli-surface.md §3.1).
   - New flags: `--revision <key>` (list executions under one revision
     via `revisions/<key>/executions/`), `--exec-id <key>` (resolve via
     `indexes/executions/<key>.json`), `--all` (list every execution
     across all revisions via `indexes/executions/`). Existing flags
     (`--watch`, `--json`, `--detailed`) unchanged.
2. `cmd/orun/command_logs.go`
   - Resolution order per cli-surface.md §4: (1)
     `--revision <revKey> --exec-id <execKey>` exact lookup, (2)
     `--exec-id <execKey>` via `indexes/executions/`, (3) default →
     latest execution from `refs/latest-execution.json`. Per-job /
     per-step targeting flags (`--job`, `--step`) unchanged. Legacy
     `.orun/executions/<id>/logs/` is read transparently when the new
     layout has no match.
3. `cmd/orun/command_describe.go`
   - New aliases (cli-surface.md §5.1):
     `orun describe revision latest`, `orun describe revision <revKey>`,
     `orun describe trigger latest`, `orun describe trigger <triggerName>`,
     `orun describe execution <execKey>`. Each renders a structured
     detail view sourced from the relevant JSON documents under the new
     layout.
   - Existing `orun describe run latest` / `orun describe run <execID>`
     continue to work and now resolve via the new execution resolver
     with legacy fallback (cli-surface.md §5.2).
   - Output for executions includes the triplet (revisionKey,
     executionKey, legacyExecID) where applicable.
4. `cmd/orun/command_get.go`
   - `orun get plans` default output: when the new layout has revisions,
     render the revision/trigger/plan/jobs/latest-exec/status table per
     cli-surface.md §6.1; when only legacy plans exist, render the legacy
     plan-hash table. `--json` returns structured output keyed
     appropriately.
5. Bridge-mirror-failed surfacing: when the read commands traverse a
   revision/execution that has `events/` containing
   `bridge-mirror-failed` records, emit a one-line stderr warning per
   distinct execution (best-effort; never block, never fail the command).
   Format `bridge-mirror-failed` warnings consistently across all four
   commands; add a tiny shared helper if useful.
6. New tests (`cmd/orun/command_status_test.go`,
   `command_logs_test.go`, `command_describe_test.go`,
   `command_get_test.go` — extend or split as needed) covering: fresh
   revision-first workspace happy path; legacy-only workspace fallback;
   mixed workspace (revisions + leftover `.orun/executions/<id>/`);
   `bridge-mirror-failed` stderr surfacing; the new `--revision`,
   `--exec-id`, `--all`, and `describe revision/trigger/execution`
   aliases.
