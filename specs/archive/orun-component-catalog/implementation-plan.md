# Implementation Plan

Milestone-based plan, modeled on `specs/orun-state-redesign/implementation-plan.md`.
Each milestone states its **goal**, its **dependencies**, its **suggested PR
scope**, and its **"done when" criteria**. Implementer agents have latitude
to merge or split milestones across PRs as long as:

- Every PR stays reviewable (one coherent change, scoped tests).
- Dependencies are respected (don't ship Milestone D before Milestone B).
- Each PR's report cites the milestone IDs it satisfies and the design
  sections it implements.

The Orchestrator's job is to pick the next ready milestone, write the task
prompt, and update `/ai/state.json`'s `active_milestone` field.

Phase 1 (`specs/orun-state-redesign/`) milestones M0–M6 are **prerequisites**
and are already complete; this plan begins at C0.

## Milestone C0 — Spec foundation and pure data models

**Goal:** ship the spec pack and the no-FS data models that every later
milestone depends on. No CLI changes.

- Land this spec set under `specs/orun-component-catalog/` (this PR).
- Add `internal/catalogmodel` package:
  - `source_snapshot.go`, `catalog_snapshot.go`, `component_manifest.go`,
    `component_yaml.go`, `graph.go`, `events.go`, `refs.go`, `indexes.go`,
    `entity_ref.go`, `sanitize.go`.
  - JSON Schema for `component.yaml` generated by `go generate ./...`,
    committed under `internal/catalogmodel/schema/`.
- Add `internal/sourcectx` package skeleton (types only, no resolver yet):
  - `model.go`, `keys.go`, `hash.go`.
- Extend `make test-state-redesign` to also gate the new packages (initially
  via empty test files — populated in C1+).

**Dependencies:** Phase 1 complete.

**Suggested PR scope:** 1–2 PRs (spec + models).

**Done when:**
- `go build ./...` and `go test ./...` are green.
- `internal/catalogmodel` decode/encode roundtrip tests pass for every
  schema in `data-model.md` (golden fixtures under `testdata/`).
- `internal/catalogmodel.Sanitize*` has 100 % coverage.
- No imports from `internal/catalogmodel` to any other `internal/` package.

**Design sections:** `data-model.md`, `identity-and-keys.md` §1–§6.

## Milestone C1 — `internal/sourcectx` resolver

**Goal:** turn a workspace into a `SourceSnapshot` deterministically.

- Implement `ResolveSourceSnapshot(opts)`:
  - Git HEAD detection (commit, ref, branch).
  - `treeHash` via `git rev-parse HEAD^{tree}` (short).
  - `dirtyHash` per `identity-and-keys.md` §7 (catalog-relevant files only).
  - `catalogInputHash` per §8.
  - `sourceScope` detection (branch/PR/tag/local-dirty/local-nogit/ci-event).
  - `SourceSnapshotKey` formatter.
- Inject Git via a `Git` interface (in-process go-git or `git` CLI shell-out
  — implementer's choice, document the trade-off in the PR).
- Inject `Clock` and `Filesystem` for testability.

**Dependencies:** C0.

**Suggested PR scope:** 1 PR (~600 LOC including tests).

**Done when:**
- `internal/sourcectx` ≥ 90 % coverage.
- Property test: `SourceSnapshotKey` is stable for the same `(commit, tree,
  dirty)` triple across 1 000 random orderings of dirty inputs.
- Clean / dirty / no-git / PR-injected / branch fixtures all produce the
  expected key.
- `--from-ci`-style provider injection produces the documented error envelope
  on no-match (mirrors Phase 1 §11).

**Design sections:** `design.md` §6, `identity-and-keys.md` §2 + §7 + §8.

## Milestone C2 — Component discovery and manifest resolver

**Goal:** turn discovered `component.yaml` files into validated
`ComponentManifest[]` with provenance, in-memory only.

- Implement `internal/catalogresolve`:
  - `discover.go` — workspace walk respecting `intent.yaml`
    `catalog.discovery.exclude`.
  - `load.go` — YAML parse, schema validation against the embedded JSON
    Schema.
  - `inherit.go` — precedence ladder per `resolution-pipeline.md` §3.
  - `infer.go` — gated by `catalog.inference.*` flags; never panics; logs
    `ErrInferenceFailed` and skips.
  - `dependencies.go` — cross-component, API, resource ref resolution.
  - `validate.go` — typed errors per `resolution-pipeline.md` §6 + §8.
  - `hash.go` — `manifestHash` per `identity-and-keys.md` §10.

**Dependencies:** C0, C1.

**Suggested PR scope:** 2 PRs (discover + load + inherit; infer + deps +
validate + hash). Implementer may also ship as one larger PR if
< ~1 200 LOC and reviewable.

**Done when:**
- `internal/catalogresolve` ≥ 90 % coverage.
- Component fixture in `testdata/repo/` produces a byte-identical
  `[]ComponentManifest` across two consecutive `Resolve` calls (T-RES-1).
- Inheritance provenance is populated for every inherited / inferred field
  (T-RES-2).
- Broken dependency reports `ErrDependencyMissing` with both endpoints.
- Cycle in `deploy-after` edges aborts; cycle in `calls` warns by default.

**Design sections:** `resolution-pipeline.md` (entire), `data-model.md` §3 +
§6 + §7.

## Milestone C3 — `CatalogSnapshot` and graph builder

**Goal:** assemble the resolved catalog from C2 outputs.

- Add `internal/catalogresolve/graph.go` — build `dependencies`, `systems`,
  `apis`, `resources`, `owners` graphs.
- Add `internal/catalogresolve/resolver.go` — top-level `Resolve(ctx, opts)`
  returning `ResolvedCatalog`.
- `catalogHash` per `identity-and-keys.md` §9.
- `summary.*` counts derived from sorted collections.

**Dependencies:** C2.

**Suggested PR scope:** 1 PR.

**Done when:**
- Same source + same inputs ⇒ same `catalogHash` (T-IDK-1).
- Changing one component's `metadata.owner` changes `catalogHash`.
- Changing only `resolution.inheritedFrom` does **not** change
  `manifestHash`.
- Graph files are deterministic byte-for-byte across runs.

**Design sections:** `resolution-pipeline.md` §1 + §7, `identity-and-keys.md`
§9 + §10.

## Milestone C4 — `internal/catalogstore` writer + resolver

**Goal:** persist `ResolvedCatalog` via `internal/statestore` with the
write-order contract.

- `internal/catalogstore/paths.go` — every helper in `catalog-store.md` §2.
- `internal/catalogstore/writer.go` — `WriteSourceSnapshot`,
  `WriteCatalogSnapshot`, `WriteRefs`, `WriteGlobalIndexes`,
  `AppendComponentEvent`.
- `internal/catalogstore/resolver.go` — read-side with the fallback chain
  in `catalog-store.md` §4.
- `internal/catalogstore/refs.go` — typed ref reader/writer.
- `internal/catalogstore/indexes.go` — typed index updates with
  `CompareAndSwap` retry.

**Dependencies:** C3.

**Suggested PR scope:** 2–3 PRs (paths + writer; resolver + fallback;
refs + indexes).

**Done when:**
- `internal/catalogstore` ≥ 90 % coverage.
- All writes go through `internal/statestore` (verified by an
  `import-restriction` lint in CI).
- Refs `current` / `main` / `branches/<x>` / `prs/<n>` all round-trip.
- Reader fallback is exercised by a test that scrubs the global index and
  asserts walk-based recovery.
- `RebuildIndexes` produces byte-identical index files for a given tree
  (T-STORE-3).

**Design sections:** `catalog-store.md` (entire).

## Milestone C6 — `orun plan` integration

**Goal:** plans live under `(SourceSnapshot, CatalogSnapshot)` parents
without breaking existing flows.

- Extend `cmd/orun/plan.go` to call `internal/sourcectx` + the resolver.
- Wire `internal/revision` to write under
  `sources/<srcKey>/catalogs/<catKey>/revisions/<revKey>/` while keeping the
  `stateCompatibilityWrites` alias.
- Plumb `--no-catalog-refresh`, `--catalog-source`, `--catalog-snapshot`,
  `--catalog-strict` flags.
- Update `PlanRevision` shape with the additive fields from
  `data-model.md` §10.

**Dependencies:** C5.

**Suggested PR scope:** 1–2 PRs (resolver wiring + revision repath;
flag + metadata).

**Done when:**
- `orun plan` produces the source/catalog/revision hierarchy.
- `plan.metadata.source` and `plan.metadata.catalog` are populated.
- Phase 1 `orun get plans`, `orun describe revision latest`, and
  `cmd/orun/state_e2e_test.go` still pass unchanged (compat suite).
- `--no-catalog-refresh` writes a revision with empty source/catalog keys
  and `metadata.catalog.skipped = true`.

**Design sections:** `design.md` §7, `compatibility-and-migration.md` §1–§5.

**Dependencies:** C6.

**Suggested PR scope:** 1 PR.

**Done when:**

**Design sections:** `design.md` §7, `compatibility-and-migration.md` §1–§4.

**Design sections:** `sync-model.md` §1–§3, §8.
