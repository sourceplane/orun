## What orun is

`orun` is a **planner** and a **cockpit** for the platform-engineering layer that sits
between application repos and deployment automation. It is shaped by four ideas:

<div className="signalGrid">
  <article className="signalCard">
    <strong>desired state</strong>
    <h3>Intent, not scripts</h3>
    <p>
      Platform teams declare environments, group policies, discovery roots, and trigger
      bindings in <code>intent.yaml</code>. App teams declare component types and inputs in
      <code>component.yaml</code>. Neither side owns the runtime details.
    </p>
  </article>
  <article className="signalCard">
    <strong>execution contract</strong>
    <h3>Compositions as packages</h3>
    <p>
      A composition is a typed, versioned execution contract — input schema, job templates,
      profiles. Distribute them as a directory, an archive, or an OCI artifact. The plan
      pins the resolved digest in <code>compositions.lock.yaml</code>.
    </p>
  </article>
  <article className="signalCard">
    <strong>compiled artifact</strong>
    <h3>The plan is the audit trail</h3>
    <p>
      Every implicit default, policy merge, and dependency edge becomes explicit in
      <code>plan.json</code>. You diff plans in pull requests, archive them as deployment
      records, and replay them against different runners.
    </p>
  </article>
  <article className="signalCard">
    <strong>operator surface</strong>
    <h3>Cockpit-grade observability</h3>
    <p>
      The same view-model that paints <code>orun status</code> drives the TUI cockpit and
      future surfaces. State is on disk under <code>.orun/</code>; <code>orun status
      --watch</code> and the TUI subscribe to the same polling stream.
    </p>
  </article>
</div>

## The compile → cockpit → execute loop

```text
intent.yaml ── component.yaml ── compositions ──┐
                                                ├──▶  orun plan  ──▶  plan.json
                  trigger context ──────────────┘                       │
                                                                        │
                                            ┌───────────────────────────┘
                                            ▼
                                       orun run  ──▶  .orun/ state
                                            │
                            ┌───────────────┼───────────────┐
                            ▼               ▼               ▼
                       orun status     orun logs       orun tui
                            └─────── cockpit (one design system) ───────┘
```

The boundary on the left is **planning**. The boundary on the right is **operations**.
`plan.json` and `.orun/` are the only contracts between them — both versioned, both
deterministic, both reviewable.

## Why this design

<div className="cardGrid">
  <article className="featureCard">
    <h3>Determinism is non-negotiable</h3>
    <p>Identical inputs produce identical plans — byte-for-byte. The planner is a pure
    function of intent + components + locked composition digests.</p>
  </article>
  <article className="featureCard">
    <h3>Policy at compile time</h3>
    <p>Group policies and domain constraints are enforced when the plan is built, not when
    it runs. A non-compliant intent fails fast with a structured error.</p>
  </article>
  <article className="featureCard">
    <h3>One design language, many surfaces</h3>
    <p>Glyphs, palette, status pills, progress bars and tree connectors live in
    <code>internal/cockpit/style</code>. Reskinning the cockpit is one file.</p>
  </article>
  <article className="featureCard">
    <h3>OCI-native distribution</h3>
    <p>orun ships as a <a href="https://github.com/sourceplane/kiox">kiox</a> provider at
    <code>ghcr.io/sourceplane/orun</code>. Compositions distribute the same way.</p>
  </article>
</div>
