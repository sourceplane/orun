## Constraints

1. The TUI service layer must call Orun internal packages directly and must not shell out to the `orun` binary. Keep `internal/tui/**` free of `exec.Command`, `os/exec`, and subprocess invocation of `orun`.
2. Preserve the plan-first safety model: `d` may start only a dry-run from an already reviewed generated plan. Real run (`r` / confirmation / apply) remains out of scope.
3. Default to fail-closed errors for unsupported execution modes. Do not accidentally enable destructive execution from the TUI.
4. Keep RunPlan event sending non-blocking or cancellation-aware so runner hooks cannot deadlock the runner if the UI stops reading.
5. Context cancellation must be honored: if `ctx` is cancelled before or during run startup, return/terminate with a clear cancelled event/error path and close the channel.
6. Keep changes bounded to `internal/tui/**` plus minimal imports/tests. Avoid changing public CLI semantics.
7. Do not log secrets. Run events may include job ids, component names, env, and error text; do not expose env var values or command output blobs in TUI event fields.
