## Objective
Complete Milestone C3 by adding the post-resolution stages — graph
construction (stage 11), `catalogHash` computation (stage 12),
`CatalogSnapshot` assembly (stage 13) — on top of the existing
`Resolve` pipeline. Produce a deterministic `(*catalogmodel.CatalogSnapshot,
[]*catalogmodel.CatalogGraph)` view from the resolved manifest set with
`summary.*` counts derived from sorted collections, byte-stable graph
files, and `catalogHash` per `identity-and-keys.md` §9. Lands T-IDK-1
(catalogHash determinism + ownership-edit sensitivity) per
`test-plan.md` §2.

## PR Boundary
- **One PR.** Scope:
  1. New file `internal/catalogresolve/graph.go` building the five
     `CatalogGraph` siblings (`dependencies`, `systems`, `apis`,
     `resources`, `owners`) from the resolved `[]*ComponentManifest`.
     Each graph's nodes and edges sorted deterministically (nodes by
     `key`; edges by `(from, to, type)` then `optional`).
  2. New file `internal/catalogresolve/catalog_hash.go` computing
     `catalogHash` per `identity-and-keys.md` §9 inputs:
     `(catalogInputHash, sorted (componentKey, manifestHash) pairs,
     canonical encoding of each CatalogGraph in fixed order
     [dependencies, systems, apis, resources, owners],
     resolver.resolverVersion)` using the canonical encoder in
     `internal/catalogmodel`.
  3. New file `internal/catalogresolve/catalog_snapshot.go` assembling
     a `*catalogmodel.CatalogSnapshot` with `summary.*` counts
     (`components`, `systems`, `apis`, `resources`, `owners`,
     `domains`) computed from sorted distinct collections, populated
     `objects.components[*]` (key, name, path, manifestHash) ordered
     by componentKey, `resolver` block fed from the existing
     `Options`-resolved namespace/repo plus an injected
     `ResolverInputs` struct (orunVersion, schemaVersion,
     resolverVersion, stackSources), and `catalogSnapshotKey` derived
     from `catalogHash` short prefix via
     `catalogmodel.FormatCatalogSnapshotKey`. `catalogSnapshotId` is a
     fresh `cat_<ulid>`.
  4. Extend `ResolvedCatalog` (or add a sibling `ResolvedCatalogFull`
     — implementer's call, keep `Resolve` exported and unchanged) so
     callers of the new builder can request the snapshot+graphs view
     **additively**. Suggested shape: a new exported
     `BuildCatalog(ctx, opts, ResolverInputs) (*CatalogView, []ValidationIssue, error)`
     where `CatalogView { ResolvedCatalog; Snapshot *CatalogSnapshot;
     Graphs []*CatalogGraph }`. Existing `Resolve` callers continue
     to work unchanged.
  5. `ResolverInputs` struct (resolverVersion, orunVersion,
     schemaVersion, stackSources, sourceSnapshotKey, catalogInputHash,
     repo, sourceScope, headRevision, treeHash, workingTree,
     authoritative, preview, createdAt) is **caller-supplied**. The
     resolver does not invent values it cannot know (notably
     `authoritative` / `preview` policy and the `sourceSnapshotKey` —
     those come from `internal/sourcectx` outputs at the call site).
     Adapter shape mirrors C1's `WorkspaceState` → resolver input
     pattern.
  6. Tests covering: T-IDK-1 (1000 random orderings of input bundles
     ⇒ identical `catalogHash`), `metadata.owner` edit changes
     `catalogHash` (the C3 acceptance signal), provenance edit
     (`resolution.inheritedFrom`) does NOT change `manifestHash` AND
     does NOT change `catalogHash` (since `manifestHash` is the only
     manifest input), graph files byte-stable across 2 consecutive
     `BuildCatalog` calls, `summary.*` counts match sorted distinct
     enumeration, `catalogSnapshotKey` regex-matches
     `^cat-[a-f0-9]{6,16}$` and is byte-stable.

## Constraints
1. **Determinism is law.** All hashing inputs flow through the
   canonical encoder in `internal/catalogmodel`; no
   `encoding/json` defaults for hashed payloads. No
   `time.Now()` inside the hash path; `createdAt` is supplied via
   `ResolverInputs`.
2. **No FS writes.** C3 is pure data only — graph and snapshot
   structs returned in memory. The C4 writer consumes these
   verbatim.
3. **Additive only.** New files preferred; do not rename or
   reshape existing exports in `internal/catalogresolve/` or
   `internal/catalogmodel/`. No renames of Phase 1 fields.
4. **Resolver inputs explicit.** `ResolverInputs` is fully
   caller-supplied; the resolver does not invent
   `authoritative`, `preview`, `sourceSnapshotKey`,
   `catalogInputHash`, `headRevision`, `treeHash`, or
   `workingTree`. Missing required fields ⇒ typed error.
5. **Sorted everything.** Nodes by `key`; edges by
   `(from, to, type, optional)`; `objects.components` by `key`;
   `summary.*` enumerations from sorted distinct sets.
6. **Coverage floors held byte-for-byte.** No probabilistic
   coverage — if rapid generators leave deterministic gaps in
   new code paths, pin them via deterministic sub-tests
   (mirror the `TestCanonicalEncodeStringEscapeBranches`
   pattern from C1's verifier-attached fix).
7. **No HTTP, no SaaS, no DB schema, no auth.** Phase 2 stays
   local-only. `internal/catalogsync` remains untouched.

## Integration Notes
- `ResolverInputs.SourceSnapshotKey` and
  `ResolverInputs.CatalogInputHash` come from
  `sourcectx.ResolveSourceSnapshot(...)` →
  `WorkspaceState.SourceSnapshotKey` and
  `sourcectx.CatalogInputHash(...)` (already shipped at C1). The
  C3 builder does NOT call `sourcectx` directly — that's the
  caller's job. This keeps the C3 surface pure data and lets C4 /
  CLI stitch the two together.
- `summary.domains` count is derived from
  `metadata.domain` (a top-level Phase 2 manifest field already
  defined in `internal/catalogmodel/component_manifest.go` —
  confirm before assuming the field name; if it's named
  differently, mirror the actual field).
