## Objective

Start implementing the Orun Cockpit TUI with a production-grade Phase 1 foundation: a registered `orun tui` Cobra command, Charm stack dependencies, the `internal/tui` package scaffold, a typed `OrunService` seam with mock/live implementations, and a minimal read-only Bubble Tea shell that loads workspace data without shelling out to `orun` subprocesses.

This implements the first coherent implementation slice from `.kiro/specs/orun-tui-cockpit/tasks.md` Phase 1 and satisfies the architecture rule that the TUI is a component-native control plane over internal Orun packages, not a CLI wrapper.

## PR Boundary

One PR should include:

1. TUI dependencies and command entry point:
   - Add pinned direct dependencies required by the spec: `github.com/charmbracelet/bubbletea`, `github.com/charmbracelet/bubbles`, `github.com/charmbracelet/lipgloss`, and `github.com/flyingmutant/rapid`.
   - Add `cmd/orun/command_tui.go` and register `orun tui` from `commands_root.go`.
   - Implement `--remote-state` and `--backend-url` validation so `--remote-state` without a backend URL fails before launching Bubble Tea.
2. TUI package foundation:
   - Create `internal/tui/app.go`, `model.go`, `keymap.go`, `theme.go`.
   - Create `internal/tui/services/` with `OrunService`, data/message types, `MockOrunService`, and `LiveOrunService` skeleton.
   - Create initial `internal/tui/views/` and `internal/tui/events/` packages as needed for a compiling shell.
3. Read-only service slice:
   - Implement `LiveOrunService.LoadWorkspace` using `internal/discovery`, `internal/loader`, `internal/normalize`, and `state.Store.ListPlans`.
   - Implement `ListRuns` against local `state.Store.ListExecutions` for the local backend path.
   - Implement a conservative non-following `TailLogs` path for local `.orun/executions/{execID}/logs/{jobID}/` logs, or leave it as a documented stub only if that is necessary to keep the PR coherent; do not fake remote behavior.
4. Minimal Bubble Tea shell:
   - `orun tui` launches Bubble Tea with alt-screen and mouse cell motion.
   - The root model asynchronously loads the workspace on init, shows loading/error states, renders a simple three-panel shell/status/key-hint view, and supports global quit/reload/focus/help basics.
   - The first UI may be minimal; do not attempt all Browse filters, Plan Studio, Run Dashboard, Log Explorer, or History interactions in this PR.
5. Tests and report:
   - Add focused unit tests for service types/mock service, remote-state URL validation, command registration, and root model basics where practical.
   - Write `ai/reports/task-0144-implementer.md` with the real PR number and validation evidence.
