Release notes for orun v2.17.0.

This is a cockpit release. The Catalog introduced in v2.16.0 graduates from a
third tab into **the cockpit's home surface**: it absorbs the Browse and
Component views, carries the full work surface on every component entity
(change overlay, run, compose, execution history), and gets the stability and
polish work a daily-driver deserves — a rebuilt live-log pipeline that stays
glitch-free under heavy streaming, a two-line header with the workspace's
source context, and an animated brand mark.

## Highlights

- **The Catalog is the cockpit.** Two top-level surfaces remain: Catalog
  (key `1`, the home surface, opens on the Component tab) and Activity
  (key `2`); `tab` toggles between them. The old Browse and Component views
  are gone — their features moved into the Catalog rather than being
  duplicated beside it.
- **Work where the knowledge is.** Component entities in the Catalog carry
  the changed/affected overlay (`CHG` column, `c` filters to changed-only),
  the last-run status (`LAST` column), and — on the detail page — the source
  detail (path, profile, watches), live change state, and an **Executions**
  section. `⏎` on an execution drops straight into the Activity
  run → jobs → steps → logs drilldown; `r` runs the component for the
  selected environment (confirm-then-execute); `g` composes it in Plan
  Studio; `e` cycles environments.
- **Stable live logs.** Watching a live job's step logs while scrolling could
  corrupt the frame (doubled headers/footers, ghost rows). The live-log
  pipeline was rebuilt on stable foundations: generation-tagged batched
  delivery, a bounded scrollback, a deterministic frame layout, and
  single-width glyphs throughout.
- **A header that knows where you are.** The header is now a two-line block:
  the breadcrumb line plus a workspace-context line — repo, branch, source
  head, a tree clean/dirty pill, the loaded catalog key, and the entity
  count. The `◆ orun` wordmark shimmers with a violet gradient sweep.

## The Catalog is the home surface

The v2.16.0 cockpit had three parallel views of the same objects — Browse,
the Component page, and the Catalog's Component tab — each with its own drill
grammar. v2.17.0 collapses them into one:

- The cockpit opens on the **Catalog's Component tab** (owner, type,
  last-run, and change columns). Later refreshes never move your tab.
- The entity detail page is the component page now: the full envelope
  (owner with provenance, stage, tier, envs), the source detail (path,
  profile, watches), walkable **Connections**, live change state, and recent
  **Executions** — one drill grammar from list to entity to run to logs.
- The list header shows the changed count; `c` toggles changed-only — the
  cockpit view of `orun catalog affected`.

Key changes that follow from the consolidation:

| Was (v2.16.0) | Now (v2.17.0) |
| --- | --- |
| `1` Components · `2` Activity · `3` Catalog | `1` Catalog · `2` Activity |
| `tab` cycles three surfaces | `tab` toggles Catalog ⇄ Activity |
| `enter` on a component opens the Component page | `enter` opens the entity detail (same content, plus the graph) |
| `o` opens the classic Component page from the Catalog | retired — the detail page *is* the component page |
| `goto.browse` palette command | aliases to the Catalog |

## Live step logs: rebuilt for stability

Streaming a live job's logs while scrolling could desync the renderer and
leave ghost rows on screen. Four root causes were fixed with standard,
stable techniques:

- **Generation-tagged streams.** Every log tail gets a stream id; batches
  from a superseded tail (a stale pump, or a cancelled stream's
  close-sentinel) are discarded whole. Moving between steps mid-run can no
  longer leak another run's lines into the view or falsely mark a live
  stream as ended.
- **Coalesced batches.** Bursts of log lines (a package install can emit
  hundreds between frames) are drained into a single message — one rebuild
  and one render per burst instead of one per line.
- **Bounded scrollback.** The buffer caps at 4,000 lines, trimming from the
  head and reporting the trim in the footer.
- **Deterministic frames.** The viewport height derives from the actual
  chrome around it, so the composed frame is exactly terminal-height in
  every streaming state — and the glyph set sticks to single-width
  characters that can never wrap a full-width line.

## Cockpit polish

- **Two-line header.** Line one is the familiar breadcrumb (intent, env,
  plan, run, mode, drill path); line two shows `repo · branch · head ·
  tree clean|dirty · catalog <key> · N entities`, omitting whatever doesn't
  exist. The brand mark animates with a violet gradient sweep — color-only,
  so it can never destabilize a frame.
- **Shared internals.** The views share one utility layer (padding,
  truncation, viewport windows — all rune- and ANSI-safe), back/esc
  drill-popping is unified across surfaces (`⌫`/`⌃o` now reliably pop one
  drill level), and entity-kind glyphs live in the shared cockpit style so
  CLI surfaces can reuse the same alphabet.

## Compatibility

- **CLI surface unchanged.** No command, flag, or output-format changes;
  this release is entirely within `orun tui` and its rendering internals.
- **TUI keymap changes** (see the table above): `1` now opens the Catalog,
  `3` is retired, and the Browse/Component views are removed. Saved
  preferences (`~/.orun/cockpit.json`) carry over unchanged.
- Catalog ids, the resolver version, and the on-disk object model are
  untouched — no cache invalidation on upgrade.
