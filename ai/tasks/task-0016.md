## Objective

Rewire `orun plan` to the canonical revision-first flow described in
`specs/orun-state-redesign/cli-surface.md` ┬Д1 and
`compatibility-and-migration.md` ┬Д1РђЊ┬Д2. Concretely:

1. Always resolve a `TriggerOccurrence` via `internal/triggerctx`
   (`ResolveTriggerContext` / `ResolveProviderEvent` / `system.manual`
   fallback) Рђћ never gated on `--trigger`/`--from-ci`.
2. Embed `metadata.trigger` and `metadata.revision` blocks in the compiled
   plan.
3. Persist the plan via `internal/revision.WriteRevision` so the canonical
   `.orun/revisions/<revisionKey>/{trigger,plan,revision,manifest}.json`
   layout, `refs/latest-revision.json`, `refs/triggers/<name>/{latest,<scope>}.json`,
   and `indexes/revisions/<key>.json` are all written atomically.
4. Always also write the legacy compat aliases
   `.orun/plans/<checksum>.json` and `.orun/plans/latest.json` (byte-identical
   copies of `plan.json`) so every preserved workflow in
   `compatibility-and-migration.md` ┬Д1 keeps working.
5. Preserve `-o/--output` semantics: when set, the user-specified path is
   ALSO written (not in place of the canonical layout) and printed alongside
   the canonical revision path.
6. Emit the new on-success summary block from `cli-surface.md` ┬Д1.1
   (`Revision`, `Trigger`, `Jobs`, `Path`).

## Constraints

1. All writes to `.orun/` must go through `internal/revision.WriteRevision`
   (or, for compat aliases, through whatever already-public alias-write
   helper `internal/revision` exposes Рђћ see `writer_compat_test.go`). Do NOT
   open-code `os.WriteFile` against `.orun/` in `cmd/orun`.
2. Trigger resolution must route through `internal/triggerctx`. Do NOT read
   `GITHUB_*` / CI env vars directly in `cmd/orun/command_plan.go`.
3. Atomicity: a failure mid-write must leave the workspace either at the
   prior revision state OR at the new revision state Рђћ never half-written.
   Rely on the `internal/statestore` atomic-write guarantees that
   `internal/revision.WriteRevision` already uses.
4. Backwards compatibility: every command sequence that worked on `main`
   before this PR must still work after it. The compat-alias contract is
   non-negotiable for M5.a.
5. `revisionKey` derivation must match `data-model.md` ┬Д3 exactly
   (`rev-<triggerScope>-<shortSHA>-p<shortPlanHash>`). Reuse the
   `internal/revision` helper that already produces this key Рђћ do not
   re-derive in `cmd/orun`.
6. JSON output must be deterministic and stable across re-runs of the same
   intent on the same SHA Рђћ `WriteRevision` already provides this; do not
   undermine it.
7. No plaintext secrets in any persisted JSON or in CLI output. The trigger
   block is sourced from `triggerctx`; do not enrich it with raw env in
   `cmd/orun`.
8. If you discover a contract gap (e.g. `internal/revision` is missing a
   public seam needed for compat-alias writes alongside `WriteRevision`),
   prefer a small additive change in `internal/revision` over working
   around it in `cmd/orun`. Justify it in the implementer report and add
   unit tests at the package boundary.

## Integration Notes

  1. Load intent (existing path).
  2. `triggerCtx, err := triggerctx.ResolveTriggerContext(...)` Рђћ pass the
     working dir + flag inputs; default to `system.manual`.

     populated from `triggerCtx` + computed plan hash.

     in `writer_compat_test.go`.
